# 规范化设计总览

## 概述

本文档定义MallEco API项目的全面规范化设计标准，确保代码质量、架构一致性和可维护性。

**项目阶段**: 开发中  
**兼容性要求**: 必须保持向后兼容  
**更新日期**: 2026-02-27

---

## 一、API设计规范

### 1.1 RESTful风格

| 操作       | 方法   | URL模式             | 示例              |
| ---------- | ------ | ------------------- | ----------------- |
| 查询列表   | GET    | /{resource}         | GET /users        |
| 查询单个   | GET    | /{resource}/:id     | GET /users/1      |
| 创建       | POST   | /{resource}         | POST /users       |
| 完整更新   | PUT    | /{resource}/:id     | PUT /users/1      |
| 部分更新   | PATCH  | /{resource}/:id     | PATCH /users/1    |
| 删除       | DELETE | /{resource}/:id     | DELETE /users/1   |
| 自定义查询 | GET    | /{resource}/:action | GET /users/export |

### 1.2 DTO验证规范

```typescript
// 基础DTO结构
export class CreateUserDto {
  @IsString()
  @MinLength(2)
  @MaxLength(50)
  name: string;

  @IsEmail()
  email: string;

  @IsEnum(UserRole)
  role: UserRole;
}
```

**规则**:

- 创建DTO以`Create`为前缀
- 更新DTO以`Update`为前缀
- 查询DTO以`Query`为前缀
- 列表DTO以`List`为前缀

### 1.3 响应格式

```typescript
// 成功响应
{
  "code": 200,
  "message": "success",
  "data": { ... },
  "timestamp": "2026-02-27T10:00:00Z"
}

// 分页响应
{
  "code": 200,
  "message": "success",
  "data": {
    "list": [...],
    "pagination": {
      "page": 1,
      "pageSize": 10,
      "total": 100
    }
  }
}

// 错误响应
{
  "code": 400,
  "message": "参数错误",
  "errors": [{ "field": "email", "message": "邮箱格式不正确" }],
  "timestamp": "2026-02-27T10:00:00Z"
}
```

### 1.4 错误码规范

格式: `{模块前缀}{三位数字}`

| 模块 | 前缀 | 范围    |
| ---- | ---- | ------- |
| 通用 | GLO  | 000-099 |
| 用户 | USR  | 100-199 |
| 订单 | ORD  | 200-299 |
| 商品 | GDS  | 300-399 |
| 支付 | PAY  | 400-499 |
| 物流 | LOG  | 500-599 |
| 认证 | AUTH | 600-699 |
| 文件 | FILE | 700-799 |

## 二、代码架构规范

### 2.1 目录结构

```
src/
├── modules/              # 业务模块
│   └── {module}/
│       ├── dto/          # 数据传输对象
│       ├── entities/    # 实体
│       ├── interfaces/  # 接口定义
│       ├── {module}.controller.ts
│       ├── {module}.service.ts
│       └── {module}.module.ts
├── shared/               # 公共基础设施
│   ├── dto/             # 公共DTO
│   ├── decorators/      # 自定义装饰器
│   ├── filters/         # 异常过滤器
│   ├── interceptors/    # 拦截器
│   ├── guards/          # 守卫
│   ├── pipes/           # 管道
│   └── utils/           # 工具函数
├── config/              # 配置管理
├── common/              # 公共常量/枚举/接口
│   ├── constants/      # 常量
│   ├── enums/          # 枚举
│   ├── interfaces/     # 全局接口
│   └── types/          # 类型定义
└── main.ts
```

### 2.2 模块规范

- 每个模块目录结构必须包含: controller, service, module
- DTO、Entity应放在模块内部，公共的放shared
- 模块间单向依赖，禁止循环依赖

## 三、数据模型规范

### 3.1 实体基类

```typescript
import {
  PrimaryGeneratedColumn,
  CreateDateColumn,
  UpdateDateColumn,
  DeleteDateColumn,
} from 'typeorm';

export abstract class BaseEntity {
  @PrimaryGeneratedColumn('increment', { type: 'bigint' })
  id: string;

  @CreateDateColumn({ name: 'created_at' })
  createdAt: Date;

  @UpdateDateColumn({ name: 'updated_at' })
  updatedAt: Date;

  @DeleteDateColumn({ name: 'deleted_at' })
  deletedAt: Date;
}
```

### 3.2 命名规范

| 类型           | 规则       | 示例         |
| -------------- | ---------- | ------------ |
| 数据库表       | 下划线命名 | user_info    |
| 数据库字段     | 下划线命名 | created_at   |
| TypeScript类   | PascalCase | UserInfo     |
| TypeScript属性 | camelCase  | userName     |
| 文件名         | kebab-case | user-info.ts |
| 接口           | PascalCase | IUserService |

## 四、基础设施规范

### 4.1 日志规范

```typescript
// 日志级别
enum LogLevel {
  DEBUG = 'debug',
  INFO = 'info',
  WARN = 'warn',
  ERROR = 'error',
}

// 日志格式
{
  "level": "info",
  "message": "User created",
  "context": "UserService",
  "timestamp": "2026-02-27T10:00:00Z",
  "metadata": { "userId": "1" }
}
```

### 4.2 错误处理分层

- **HTTP层**: @nestjs/platform-express 捕获
- **应用层**: 自定义异常过滤器
- **业务层**: 业务异常类
- **基础设施层**: 全局异常捕获

## 五、实施策略

### 5.1 渐进式迁移

1. **第一阶段**: 创建shared公共基础设施
2. **第二阶段**: 新模块遵循规范
3. **第三阶段**: 重构旧模块

### 5.2 兼容性策略

- 保持现有API签名不变
- 新增规范不破坏现有功能
- 逐步废弃旧模式
