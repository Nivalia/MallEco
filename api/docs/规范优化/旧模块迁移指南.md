# 旧模块迁移指南

## 概述

本文档指导如何将现有旧模块逐步迁移到新的规范化体系。迁移采用渐进式策略，确保向后兼容。

## 迁移原则

1. **渐进式**: 分批次迁移，不一次性大规模修改
2. **兼容性**: 保持API接口不变，确保前端无感知
3. **可回滚**: 每次迁移可独立回滚
4. **测试覆盖**: 迁移后必须通过测试验证

## 迁移检查清单

### 1. Controller层

| 检查项             | 状态 | 说明                                         |
| ------------------ | ---- | -------------------------------------------- |
| 使用统一响应格式   | ⬜   | 返回值使用 `return { data, message, code }`  |
| Swagger注解完善    | ⬜   | 添加 `@ApiTags`, `@ApiOperation` 等          |
| 错误处理使用异常类 | ⬜   | 使用 `BusinessException` 替代直接throw Error |
| 路由命名规范       | ⬜   | RESTful风格，资源用名词                      |

### 2. Service层

| 检查项          | 状态 | 说明                          |
| --------------- | ---- | ----------------------------- |
| 继承通用Service | ⬜   | 考虑继承 `BaseService`        |
| 使用ErrorCode   | ⬜   | 业务异常使用 `ErrorCode` 枚举 |
| 日志规范        | ⬜   | 使用 `Logger` 并设置context   |
| DTO验证         | ⬜   | 使用 `class-validator`        |

### 3. Entity层

| 检查项         | 状态 | 说明                     |
| -------------- | ---- | ------------------------ |
| 继承BaseEntity | ⬜   | 使用统一的实体基类       |
| 字段命名       | ⬜   | 数据库用下划线，TS用驼峰 |
| 软删除         | ⬜   | 使用 `DeleteDateColumn`  |

### 4. DTO层

| 检查项              | 状态 | 说明                        |
| ------------------- | ---- | --------------------------- |
| 使用class-validator | ⬜   | 添加验证装饰器              |
| Swagger注解         | ⬜   | 使用 `@ApiProperty`         |
| 继承公共DTO         | ⬜   | 列表查询继承 `ListQueryDto` |

## 迁移步骤

### 步骤1: 分析模块

检查现有模块结构，识别需要迁移的内容：

```bash
# 查看当前模块结构
ls -la src/modules/{module-name}/
```

### 步骤2: 迁移Entity

将实体改为继承BaseEntity:

```typescript
// 旧代码
@Entity()
export class User {
  @PrimaryGeneratedColumn()
  id: number;

  @CreateDateColumn()
  createTime: Date;
}

// 新代码
@Entity()
export class User extends BaseEntity {
  // 只需要添加业务字段
  @Column()
  username: string;
}
```

### 步骤3: 迁移DTO

添加验证和Swagger注解:

```typescript
// 旧代码
export class CreateUserDto {
  username: string;
  email: string;
}

// 新代码
export class CreateUserDto {
  @ApiProperty({ description: '用户名' })
  @IsString()
  @MinLength(2)
  @MaxLength(50)
  username: string;

  @ApiProperty({ description: '邮箱' })
  @IsEmail()
  email: string;
}
```

### 步骤4: 迁移Service

统一错误处理和日志:

```typescript
// 旧代码
async findOne(id: number) {
  const user = await this.userRepository.findOne(id);
  if (!user) {
    throw new NotFoundException('用户不存在');
  }
  return user;
}

// 新代码
async findOne(id: string) {
  const user = await this.userRepository.findOne({ where: { id: BigInt(id) } });
  if (!user) {
    throw BusinessException.dataNotFound('用户不存在');
  }
  return user;
}
```

### 步骤5: 迁移Controller

完善Swagger和响应格式:

```typescript
// 旧代码
@Get(':id')
async findOne(@Param('id') id: number) {
  return this.userService.findOne(id);
}

// 新代码
@ApiTags('用户管理')
@ApiOperation({ summary: '获取用户详情' })
@ApiResponse({ status: 200, description: '查询成功' })
@ApiResponse({ status: 404, description: '用户不存在' })
@Get(':id')
async findOne(@Param('id') id: string) {
  return this.userService.findOne(id);
}
```

## 迁移优先级

### 高优先级 (先迁移)

1. **核心业务模块**: 订单、商品、用户、支付
2. **高频使用模块**: 购物车、收藏、评论
3. **存在问题的模块**: 有较多bug或技术债

### 中优先级

1. **运营模块**: 促销、优惠券、内容
2. **辅助模块**: 物流、通知、文件

### 低优先级

1. **管理模块**: 系统配置、日志、监控
2. **第三方模块**: 微信、短信、邮件

## 回滚方案

如果迁移后出现问题:

1. **立即回滚**: 使用git回滚代码
2. **逐个回滚**: 按文件逐个检查
3. **对比测试**: 使用测试用例验证功能

## 常见问题

### Q: 迁移会影响现有API吗？

A: 只要不改变路由和方法签名，就不会影响。如果需要改变，先做好版本控制。

### Q: 旧模块太多，如何开始？

A: 按优先级排序，每次只迁移一个模块，完成测试后再继续。

### Q: 如何验证迁移是否正确？

A: 1. 运行单元测试 2. 使用Postman测试API 3. 对比迁移前后响应格式
